# Phase 12 — Compounding Memory Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.
> Each slice is one atomic git commit. Test-first (TDD). Frequent commits. Reference design doc at `docs/plans/2026-05-02-phase-12-compounding-memory-design.md`.

**Goal:** Make Aura's memory compound automatically — every conversation contributes durable wiki knowledge, the wiki self-maintains, and the dashboard exposes the new surfaces.

**Architecture:** Three coupled backend components (conversation archive in SQLite, post-turn auto-summarization with salience scoring + dedup, scheduled wiki maintenance with auto-fix), three matching frontend routes, comprehensive Q&A coverage. Executed by a 3-teammate Claude Code Agent Team (Backend / Frontend / Q&A) all on Sonnet 4.6, with disjoint file ownership. Lead orchestrates from main session. Final code review by Opus 4.7.

**Tech Stack:** Go 1.22+, SQLite (`database/sql`), chromem-go (search), Mistral (embeddings + LLM scoring), telebot.v4, React 19 + Vite + react-router-dom v7, shadcn/ui + Tailwind, sonner toasts, lucide-react icons.

**Conventions** (from `AGENTS.md`, `CLAUDE.md`, project memory):
- One atomic git commit per slice; stage explicit paths only (no `git add -A`).
- Tests must pass before commit. Body: `Body` (not `Content`) for wiki pages. Wiki links `[[slug]]`.
- `LLM_API_KEY` for chat; `EMBEDDING_API_KEY` separate for Mistral embeddings.
- `.env.example` always synced with `.env` for new config keys.
- Tool argument logging: only keys, never values.
- Frontend bundle rebuild commits both `web/` and `internal/api/dist/`.
